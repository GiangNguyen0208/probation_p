# Phase 3 - Telegram Mini App

Window: Weeks 4-6  
Sprints: Sprints 5-6  
Goal: analysts can monitor all subjects from inside Telegram.

## Scope

Build the analyst-facing Telegram Mini App using the API Gateway only. The Mini App should support monitoring, search, filtering, charts, summary metrics, and alert configuration.

Even with a Python backend, keep this repository as a TypeScript/React frontend. Telegram Mini Apps run in a WebView, so Python should not replace the client UI. The Python-specific work here is consuming API contracts generated from backend Pydantic models.

## Repositories

- `social-mini-app`
- `social-api-gateway`

## Sprint 5 Focus

- Initialize the React/Vite Telegram Mini App from the official template.
- Configure Telegram WebApp SDK lifecycle handling, theme variables, safe areas, and back button behavior.
- Import or reference JSON Schema/OpenAPI contracts generated by the Python API where useful for typed API clients.
- Build subject list screen with platform badge, status, follower count, post frequency, and last sync time.
- Add search and filter UI for platform, priority, and status.
- Connect list and filters to `GET /v1/subjects`.
- Add loading, empty, and API error states.

## Sprint 6 Focus

- Build subject detail screen with follower trend and post frequency charts.
- Add summary dashboard with total tracked subjects, alerts today, most active platform, and last sync timestamp.
- Build alert configuration panel for rule type, threshold, cooldown, and channel.
- Connect alert config panel to internal API endpoints.
- Test rendering and navigation inside the Telegram WebView.
- Add Mini App README with setup, Telegram bot linkage, and environment variables.

## Deliverables

- Telegram-native subject monitoring UI.
- Search and filters backed by API query params.
- Subject detail charts using activity time-series data.
- Summary dashboard.
- Alert configuration panel.
- Telegram WebApp SDK integration.
- API client aligned with FastAPI/OpenAPI response contracts.

## Acceptance Criteria

- Analysts can view and filter subject lists inside Telegram.
- Analysts can open a subject and inspect historical activity.
- Analysts can create or update alert thresholds for a subject.
- Theme, safe area, viewport, and back button behavior work in actual Telegram.
- All Mini App data access goes through the API Gateway.

## Dependencies

- Phase 2 API Gateway endpoints are available.
- Internal API key provisioning for Mini App.
- Telegram bot/app setup for WebApp launch.
- Generated API contract or documented OpenAPI schema is available from the Python gateway.

## Risks

- Telegram WebView rendering quirks can differ from normal browser behavior.
- API latency or pagination issues can make list views feel slow.
- Alert configuration UX can become error-prone if thresholds are not validated clearly.
- Contract drift can appear if frontend types are hand-written instead of generated or checked against OpenAPI.

## Sprint Exit

The phase exits when an analyst can use the Telegram Mini App as the primary monitoring surface and configure alert rules without leaving Telegram.
